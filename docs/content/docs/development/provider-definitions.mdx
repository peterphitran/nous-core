# Provider Definitions

Provider definitions live in `self/subcortex/providers/src/definitions/`. They describe provider metadata in a static typed catalog so contributors can add provider identity, protocol, default endpoint, default model, credential metadata, locality, and capability hints without touching runtime construction logic first.

The Phase 1.1 catalog covers the current provider roster:

| Definition | Vendor key | Protocol |
|------------|------------|----------|
| `anthropic.ts` | `anthropic` | `anthropic-messages` |
| `chat-completions.ts` | `openai` | `chat-completions` |
| `ollama.ts` | `ollama` | `ollama` |

## Authoring Pattern

Each provider leaf exports a metadata constant created with `defineProvider(...)`:

```ts
export const EXAMPLE_PROVIDER_DEFINITION = defineProvider({
  vendorKey: 'example',
  displayName: 'Example',
  wellKnownProviderId: '10000000-0000-0000-0000-000000000099',
  providerType: 'text',
  providerClass: 'remote_text',
  protocol: 'chat-completions',
  adapterKey: 'chat-completions',
  defaultEndpoint: 'https://api.example.com',
  defaultModelId: 'example-model',
  auth: {
    envVar: 'EXAMPLE_API_KEY',
    vaultKeyNamespace: 'example',
    required: true,
    purpose: 'api_key',
  },
  isLocal: false,
});
```

`defineProvider(...)` validates the definition with the provider-definition Zod schema and preserves literal TypeScript inference. The aggregator exports `PROVIDER_DEFINITIONS`, `ProviderVendorKey`, and `BootstrapProviderKey` from the registered `vendorKey` literals.

## Metadata-Only Rule

Definition leaf files are metadata only. They should not:

- import concrete provider classes,
- instantiate providers,
- read `process.env`,
- perform network calls, or
- contain provider-specific bootstrap branches.

Runtime behavior remains in the concrete provider classes and registry paths. Adding a new production provider still requires explicit review of adapter behavior, credential handling, tests, and bootstrap consumption in the appropriate WR-177 sub-phase.

## Current Boundaries

Phase 1.1 adds the static catalog and the clean-break rename to `ChatCompletionsProvider`. It does not make bootstrap or credential loading registry-driven yet. Adapter module relocation is planned for WR-177 Phase 1.2, and registry-derived bootstrap and credential consumption are planned for WR-177 Phase 1.3.
