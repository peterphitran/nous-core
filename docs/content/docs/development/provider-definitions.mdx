# Provider Definitions

Provider definitions live in `self/subcortex/providers/src/definitions/`. They describe provider metadata in a static typed catalog so contributors can add provider identity, protocol, default endpoint, default model, credential metadata, locality, and capability hints without touching runtime construction logic first.

The current catalog covers the provider roster:

| Definition | Vendor key | Protocol |
|------------|------------|----------|
| `anthropic.ts` | `anthropic` | `anthropic-messages` |
| `chat-completions.ts` | `openai` | `chat-completions` |
| `ollama.ts` | `ollama` | `ollama` |

## Authoring Pattern

Each provider leaf exports a metadata constant created with `defineProvider(...)`:

```ts
export const EXAMPLE_PROVIDER_DEFINITION = defineProvider({
  vendorKey: 'example',
  displayName: 'Example',
  wellKnownProviderId: '10000000-0000-0000-0000-000000000099',
  providerType: 'text',
  providerClass: 'remote_text',
  protocol: 'chat-completions',
  adapterKey: 'chat-completions',
  defaultEndpoint: 'https://api.example.com',
  defaultModelId: 'example-model',
  auth: {
    envVar: 'EXAMPLE_API_KEY',
    vaultKeyNamespace: 'example',
    required: true,
    purpose: 'api_key',
  },
  isLocal: false,
});
```

`defineProvider(...)` validates the definition with the provider-definition Zod schema and preserves literal TypeScript inference. The aggregator exports `PROVIDER_DEFINITIONS`, `ProviderVendorKey`, and `BootstrapProviderKey` from the registered `vendorKey` literals.

## Metadata-Only Rule

Definition leaf files are metadata only. They should not:

- import concrete provider classes,
- instantiate providers,
- read `process.env`,
- perform network calls, or
- contain provider-specific bootstrap branches.

Runtime behavior remains in the concrete provider classes and registry paths. Adding a new production provider still requires explicit review of adapter behavior, credential handling, tests, and bootstrap consumption before it lands.

## Adapter Modules

Provider adapter behavior lives in `self/subcortex/providers/src/adapters/`. Adapter modules use `defineProviderAdapter(...)` and expose `adapterKey`, `displayName`, `protocol`, `capabilities`, and `create(options?)`.

Current adapter keys are:

| Adapter module | Adapter key | Protocol |
|----------------|-------------|----------|
| `anthropic-adapter.ts` | `anthropic` | `anthropic-messages` |
| `chat-completions-adapter.ts` | `chat-completions` | `chat-completions` |
| `ollama-adapter.ts` | `ollama` | `ollama` |
| `text-adapter.ts` | `text` | `text-fallback` |

`openai` remains a provider vendor key for the current Chat Completions-backed definition, but it resolves to the canonical `chat-completions` adapter key.

## Current Boundaries

The provider definition catalog and adapter module registry are now provider-package owned. Bootstrap and credential loading are not registry-driven yet; registry-derived bootstrap and credential consumption remain planned follow-up work.
